# ユーザをDBで管理するための要件定義

## 1. 背景
現在の `pdf_viewer` は、検索 API とファイル配信を中心とした構成であり、ユーザ情報を永続化する仕組みがありません。
現状の認証は Nginx の Basic 認証など、ファイルベースまたは設定ベースでの運用を前提にしています。

今後は、ユーザをデータベースで管理できるようにし、ログイン・権限管理・運用変更をアプリケーション側で扱えるようにします。

## 2. 目的
- ユーザ情報を DB に保存し、永続的に管理できるようにする
- ユーザの追加・更新・無効化・削除をアプリケーションから行えるようにする
- 認証と認可を DB ベースに移行し、設定ファイル依存を減らす
- 将来的な権限追加や監査ログ追加に備える

## 3. 現状整理
### 3.1 既存構成
- API: Hono + Meilisearch 連携
- ファイル配信: Nginx
- ユーザ管理: 未実装
- データ永続化: Meilisearch とファイルストレージ中心

### 3.2 現状の課題
- ユーザ情報が設定ファイルに存在しない
- 認証対象のユーザを追加・更新するたびに手作業が発生する
- 権限を持つユーザと一般ユーザを区別できない
- 操作履歴を追跡できない

## 4. スコープ
### 4.1 対象
- ユーザアカウント管理
- ログイン認証
- 権限管理
- ユーザの有効・無効管理
- 管理者によるユーザ操作

### 4.2 対象外
- パスワードレス認証
- ソーシャルログイン
- 外部 IdP 連携
- 多要素認証
- 詳細な監査基盤の整備

## 5. 業務要件
### 5.1 ユーザ管理
- 管理者はユーザを新規作成できること
- 管理者はユーザ情報を編集できること
- 管理者はユーザを無効化できること
- 管理者はユーザを削除できること
- 管理者はユーザ一覧を検索・閲覧できること

### 5.2 認証
- ユーザは ID とパスワードでログインできること
- パスワードは平文保存しないこと
- ログイン失敗時は適切に拒否されること
- 無効化されたユーザはログインできないこと

### 5.3 認可
- ユーザごとに権限を保持できること
- 代表的な権限として管理者と一般ユーザを扱えること
- API や管理画面の操作権限を権限情報に基づいて制御できること

### 5.4 運用
- ユーザ追加・停止・再開を DB 更新で反映できること
- ユーザの状態変更が即時に認証へ反映されること
- 将来的に CSV インポートや一括更新へ拡張できること

## 6. 機能要件
### 6.1 ユーザ情報の保存
DB に以下を保存する。
- ユーザ ID
- ログイン ID
- 表示名
- メールアドレス
- パスワードハッシュ
- 権限
- 有効フラグ
- 作成日時
- 更新日時
- 最終ログイン日時

### 6.2 ユーザ作成
- ログイン ID は一意であること
- パスワードは登録時にハッシュ化すること
- 初期権限を指定できること

### 6.3 ユーザ更新
- 表示名、メールアドレス、権限、有効状態を更新できること
- パスワードのみを変更できること

### 6.4 ユーザ削除
- 論理削除を基本とすること
- 物理削除は管理者限定の運用にすること

### 6.5 ログイン
- ログイン成功時にセッションまたはトークンを発行すること
- ログイン失敗時は理由を過剰に開示しないこと
- 一定回数失敗した場合の制御を検討すること

### 6.6 認証連携
- API 側で認証済みユーザを識別できること
- Nginx ベースの Basic 認証に依存しない構成に移行できること
- 必要に応じて Nginx はリバースプロキシのみ担当すること

## 7. データ要件
### 7.1 ユーザテーブル案
| 項目          | 型            | 必須 | 説明                     |
| ------------- | ------------- | ---- | ------------------------ |
| id            | UUID / bigint | 必須 | 主キー                   |
| login_id      | string        | 必須 | ログイン用 ID、一意      |
| display_name  | string        | 必須 | 表示名                   |
| email         | string        | 任意 | メールアドレス           |
| password_hash | string        | 必須 | ハッシュ化済みパスワード |
| role          | string        | 必須 | admin / user など        |
| is_active     | boolean       | 必須 | 有効状態                 |
| last_login_at | datetime      | 任意 | 最終ログイン日時         |
| created_at    | datetime      | 必須 | 作成日時                 |
| updated_at    | datetime      | 必須 | 更新日時                 |
| deleted_at    | datetime      | 任意 | 論理削除日時             |

### 7.2 制約
- `login_id` は一意制約を持つこと
- `email` を使う場合は重複可否を要件で決めること
- パスワードは十分強いハッシュ方式を使うこと
- 監査や復旧のため、削除は論理削除を基本とすること

## 8. 権限要件
### 8.1 権限区分
- `admin`: ユーザ管理、設定変更、全データ参照
- `user`: 一般利用、限定された参照のみ

### 8.2 権限制御方針
- 管理系 API は `admin` のみ利用可とする
- 一般利用 API は必要に応じて `user` 以上で利用可とする
- 権限チェックは API 層で必ず行うこと

## 9. 非機能要件
### 9.1 セキュリティ
- パスワードは平文保存しない
- 認証情報はログに出力しない
- 管理 API は認証・認可を必須とする
- 退会済み・無効化済みユーザを利用不可にする

### 9.2 可用性
- DB 障害時の挙動を定義すること
- 最低限、認証不能時は安全側に倒すこと

### 9.3 保守性
- ユーザ管理ロジックを API と分離して実装できること
- DB スキーマ変更に備えてマイグレーションを導入できること

### 9.4 拡張性
- 将来的にグループ、組織、複数ロールへ拡張できること
- 外部認証基盤と連携できる設計余地を残すこと

## 10. 画面・API 要件
### 10.1 管理画面
- ユーザ一覧
- ユーザ詳細
- 新規作成
- 編集
- 無効化 / 有効化
- パスワード再設定

### 10.2 API 例
- `POST /auth/login`
- `POST /auth/logout`
- `GET /users`
- `POST /users`
- `GET /users/:id`
- `PATCH /users/:id`
- `DELETE /users/:id`

## 11. 運用要件
- 初期管理者を 1 件以上作成できること
- パスワード初期設定後に変更を促せること
- ユーザ無効化時は既存セッションの失効方針を定義すること
- バックアップと復元手順を定義すること

## 12. 受け入れ条件
- DB にユーザ情報を登録できる
- ログイン処理が DB のユーザ情報を参照して動作する
- 権限に応じて管理操作を制限できる
- 無効化されたユーザは利用できない
- パスワードが平文で保存されていない

## 13. 実装前提の推奨事項
- DB は PostgreSQL を第一候補とする
- ORM または migration ツールを導入する
- パスワードハッシュは bcrypt か argon2 を採用する
- API 側で認証ミドルウェアを導入する
- Nginx は認証ロジックを持たず、プロキシと TLS 終端に寄せる

## 14. 未決事項
以下は実装前に確定する必要がある。
- 使用する DB の種類
- 認証方式をセッションにするか JWT にするか
- 管理画面を別途作るか API のみで開始するか
- 既存の Basic 認証をいつまで併用するか
- 権限の粒度を admin / user だけにするか
